## 1. 개요

### 1.1 배경

현재 사주 계산 엔진(`core/` 패키지)은 사주 사기둥(四柱)과 대운(大運) 계산까지 구현되어 있다. 그러나 `saju_option.md`에 정의된 요구 기능 중 다음 항목이 아직 미구현 상태이다:

- **십이운성(十二運星)**: 일간 오행 기준으로 각 지지에 대한 12단계 생사 사이클 산출
- **신살(神殺)**: 연지·월지·일지 기반의 길흉 신살 판별
- **세운(歲運)**: 현재 연도 기준 ±5년(총 11년) 간지 산출
- **SajuResult 모델 확장**: 이미 계산되었지만 API 응답에 노출되지 않은 지장간·육신·오행비율 포함, 신규 계산 결과를 API 응답에 추가

또한 기존에 계산은 되지만 `SajuResult`에 담기지 않아 API 및 UI에 전달되지 않는 항목들(지장간, 십성/육신, 오행비율)도 이번 SPEC에서 함께 통합한다.

### 1.2 범위

본 SPEC은 다음 범위를 포함한다:

- `core/sibiunsung.py` 신규 생성: 십이운성 계산 모듈
- `core/shinsal.py` 신규 생성: 신살 계산 모듈
- `core/deun.py` 확장: 세운 계산 함수 추가
- `core/models/domain.py` 확장: 십이운성, 신살, 세운, 기둥 의미 도메인 모델 추가
- `core/models/response.py` 확장: `SajuResult`에 새 필드 추가
- `app/services/saju_service.py` 확장: 신규 계산 결과를 `SajuResult`에 조립

본 SPEC은 다음을 포함하지 않는다:

- 합충형해(hapchung.py) 신규 계산 (기존 구현 유지)
- 용신(yongshin.py) 로직 변경
- UI(Streamlit) 또는 API 라우터 시그니처 변경 (내부 응답 모델만 확장)
- 음력 변환 로직 변경

### 1.3 목표

1. 십이운성·신살·세운 계산 모듈을 신규 구현하여 테스트 커버리지 85% 이상 달성
2. `SajuResult` 응답에 지장간·육신·오행비율·십이운성·신살·세운·기둥 의미를 포함
3. 기존 API 엔드포인트(`POST /api/v1/saju`)의 하위 호환성을 유지하면서 응답 데이터를 풍부하게 확장
4. 모든 신규 계산 함수는 순수 함수(pure function) 형태로 작성하여 재사용성과 테스트 용이성을 확보

## 3. 요구사항 명세 (EARS 형식)

### 3.1 항상 적용되는 요구사항 (Ubiquitous)

**FR-001**: 시스템은 항상 십이운성 계산에 일간(日干)의 오행을 기준으로 사용해야 한다.

**FR-002**: 시스템은 항상 오행별 십이운성 시작 지지를 다음 규칙에 따라 적용해야 한다:
- 木 일간: 亥(해)에서 장생
- 火 일간: 寅(인)에서 장생
- 土 일간: 申(신)에서 장생
- 金 일간: 巳(사)에서 장생
- 水 일간: 申(신)에서 장생

**FR-003**: 시스템은 항상 십이운성의 순서를 장생→목욕→관대→건록→제왕→쇠→병→사→묘→절→태→양 순으로 적용해야 한다.

**FR-004**: 시스템은 항상 음(陰) 일간의 경우 십이운성 순서를 역방향(양→태→절→묘→사→병→쇠→제왕→건록→관대→목욕→장생)으로 적용해야 한다.

**FR-005**: 시스템은 항상 세운 계산 시 60갑자 순환 규칙을 사용해야 한다.

**FR-006**: 시스템은 항상 기존 `SajuResult` 필드(year_pillar, month_pillar, day_pillar, hour_pillar, deun)를 유지하면서 새 필드를 추가해야 한다 (하위 호환성 보장).

**FR-007**: 시스템은 항상 `SajuResult`의 신규 필드를 Optional로 선언하여 시주 미입력 시에도 정상 동작해야 한다.

### 3.2 이벤트 기반 요구사항 (Event-Driven)

**FR-008**: WHEN 사주 계산 요청이 수신되면 THEN 시스템은 일간을 추출하여 4개 기둥 각각의 지지에 대한 십이운성을 계산해야 한다.

**FR-009**: WHEN 사주 계산 요청이 수신되면 THEN 시스템은 연지·월지·일지를 기반으로 역마살, 도화살, 화개살, 백호살, 천을귀인의 해당 여부를 판별해야 한다.

**FR-010**: WHEN 사주 계산 요청이 수신되면 THEN 시스템은 현재 연도(서버 시스템 날짜 기준)를 중심으로 ±5년(총 11개)의 세운 간지를 계산해야 한다.

**FR-011**: WHEN 사주 계산 요청이 수신되면 THEN 시스템은 이미 계산된 지장간, 육신, 오행비율을 `SajuResult`에 포함시켜 반환해야 한다.

**FR-012**: WHEN 세운 목록을 계산하면 THEN 시스템은 현재 연도에 해당하는 세운 항목에 `is_current: true` 플래그를 표시해야 한다.

**FR-013**: WHEN 신살 계산 요청이 수행되면 THEN 시스템은 해당되는 신살 이름과 해당 지지를 포함한 목록을 반환해야 한다.

**FR-014**: WHEN 기둥별 의미(pillar_meanings)를 생성하면 THEN 시스템은 연주·월주·일주·시주 각각에 고정 의미 텍스트(조상/어린시절, 부모/직업환경, 나/배우자, 자녀/말년)를 매핑해야 한다.

### 3.3 상태 기반 요구사항 (State-Driven)

**FR-015**: IF 시주(时柱)가 입력되지 않은 경우 THEN 시스템은 시주에 대한 십이운성 계산을 생략하고 해당 항목을 None으로 반환해야 한다.

**FR-016**: IF 일간이 무(戊) 또는 기(己)인 경우 THEN 시스템은 토(土) 오행 기준 십이운성 표를 적용해야 한다.

**FR-017**: IF 신살 판별 기준 지지 조합이 사주에 존재하지 않는 경우 THEN 시스템은 해당 신살 항목을 결과 목록에 포함시키지 않아야 한다.

**FR-018**: IF 사주 계산 중 신규 계산 모듈(십이운성, 신살, 세운)에서 예외가 발생하는 경우 THEN 시스템은 해당 필드를 None으로 설정하고 기존 계산 결과는 그대로 반환해야 한다 (부분 실패 허용).

### 3.4 선택적 요구사항 (Optional)

**FR-019**: 가능하면 각 기둥의 십이운성에 해당 단계의 한자 표기와 설명을 함께 제공한다.

**FR-020**: 가능하면 신살 결과에 각 신살의 의미(한 줄 설명)를 포함한다.

### 3.5 금지 사항 (Unwanted)

**FR-021**: 시스템은 십이운성 계산 시 음양이 다른 일간에게 동일한 십이운성 표를 적용하지 않아야 한다 (양간과 음간은 서로 역방향).

**FR-022**: 시스템은 기존 `core/` 모듈의 공개 인터페이스(함수 시그니처)를 변경하지 않아야 한다 (확장만 허용).

**FR-023**: 시스템은 세운 계산 시 연도 기준을 클라이언트 입력값으로 받지 않아야 한다 (서버 시스템 시간 기준).

### 4.3 핵심 계산 로직 명세

#### 십이운성 (`core/sibiunsung.py`)

```
SIBI_UNSUNG_STAGES = ["장생", "목욕", "관대", "건록", "제왕", "쇠", "병", "사", "묘", "절", "태", "양"]

# 오행별 장생 시작 지지 (양간 기준)
JANGSAENG_START = {
    "목": "해",  # 亥
    "화": "인",  # 寅
    "토": "신",  # 申
    "금": "사",  # 巳
    "수": "신",  # 申
}

# 지지 순서 (자축인묘진사오미신유술해)
JI_ORDER = ["자", "축", "인", "묘", "진", "사", "오", "미", "신", "유", "술", "해"]
```

**calc_sibiunsung(day_gan: str, target_ji: str) -> str**
- 일간의 오행 추출 → 양/음 판별
- 양간: 장생 시작 지지에서 순방향으로 12단계 인덱스 계산
- 음간: 장생 시작 지지에서 역방향으로 12단계 인덱스 계산
- 반환: 해당 십이운성 단계명

#### 신살 (`core/shinsal.py`)

```
# 연지(年支) 기준 역마살 대응표
YEOKMA_MAP = {"인": "신", "신": "인", "사": "해", "해": "사", "자": "인", "오": "신", "묘": "사", "유": "해"}

# 연지 기준 도화살 대응표
DOHWA_MAP = {"인": "묘", "신": "유", "사": "오", "해": "자", "자": "유", "오": "묘", "묘": "자", "유": "오"}

# 연지 기준 화개살 대응표
HWAGAE_MAP = {"인": "술", "신": "진", "사": "축", "해": "미", "자": "진", "오": "술", "묘": "미", "유": "축"}
```

**calc_shinsal(year_ji: str, month_ji: str, day_ji: str, day_gan: str, pillars_ji: list[str]) -> list[ShinsalItem]**
- 각 신살 판별 함수 내부 호출 후 결과 집계
- 해당되는 신살만 목록에 포함

#### 세운 (`core/deun.py` 확장)

**calc_sewun(base_year: int, current_year: int) -> list[SewunItem]**
- `current_year` 기준 -5 ~ +5 범위의 11개 연도에 대해 60갑자 계산
- `SewunItem(year: int, ganji: GanJi, is_current: bool)` 반환

## 5. 제약 사항

### 5.1 성능 제약

- 신규 계산 함수(십이운성, 신살, 세운) 포함 전체 사주 계산은 기존 대비 200ms 이내 추가 지연만 허용
- 상수 조회 테이블(딕셔너리)을 모듈 레벨에서 정의하여 반복 계산 최소화

### 5.2 호환성 제약

- Python 3.11 이상 사용 (기존 환경 유지)
- Pydantic v2 모델 패턴 사용
- 기존 테스트 231개가 모두 통과해야 함 (회귀 방지)

### 5.3 품질 제약

- 신규 모듈의 테스트 커버리지 85% 이상
- ruff 린트 오류 0건
- mypy 타입 오류 0건
- 모든 공개 함수에 docstring 작성

### 5.4 설계 제약

- 모든 계산 함수는 순수 함수(pure function)로 작성 (외부 상태 참조 금지, 단 상수 테이블 제외)
- 신규 도메인 모델은 Pydantic BaseModel 상속
- 세운 계산의 현재 연도는 서비스 레이어에서 주입 (계산 함수는 `current_year: int` 파라미터 수신)
